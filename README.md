# 🚀 ShardARC-Cache

> 基于 C++14 实现的高性能、自适应、分布式内存缓存系统。

在现代高并发的后端架构中，内存缓存是缓解数据库压力、提升系统吞吐量的核心组件。**ShardARC-Cache** 旨在从零实现一个工业级的高性能内存缓存系统，解决传统 LRU 缓存的“缓存污染”问题以及多核环境下的全局锁瓶颈，并深度集成了生产环境必备的高可用防御机制。

## ✨ 核心特性

*   🧠 **高命中率 (ARC 算法)**：引入 IBM 提出的 ARC（自适应替换缓存）算法，动态平衡 LRU 与 LFU，有效应对突发稀疏流量，防止缓存污染。
*   ⚡ **极速并发 (分段锁架构)**：采用 16 分片（Sharded）分段锁设计，将全局锁竞争概率降至 1/16，突破多核 CPU 并发瓶颈。
*   🛡️ **高可用防御 (三大异常处理)**：
    *   **防缓存穿透**：内置空对象缓存（Cache Null）机制拦截非法 Key。
    *   **防缓存击穿**：采用 Singleflight（Loader Lock）与双重检查锁定（DCL），保护 DB 免受热点 Key 过期带来的并发冲击。
    *   **防缓存雪崩**：引入 TTL 随机抖动（Jitter），将过期时间均匀打散。
*   🌐 **跨语言调用 (RESTful API)**：基于 `cpp-httplib` 提供轻量级 HTTP 接口，支持 Go/Python/Java 等多语言作为独立中间件调用。
*   ⏳ **生命周期管理**：支持基于时间的 TTL 惰性删除，并提供数据淘汰时的驱逐回调（Eviction Callback）以解耦业务。

## 🛠️ 技术栈

*   **开发语言**：现代 C++ (C++14) - 充分利用 RAII、泛型编程与标准并发库
*   **核心算法**：ARC (Adaptive Replacement Cache)
*   **并发控制**：Sharded Locking（分段锁）
*   **网络通信**：`cpp-httplib`
*   **构建工具**：CMake

## 🏗️ 核心架构设计

### 1. ARC 算法自适应机制
系统内部维护四个队列（T1, T2, B1, B2）：
*   **T1 (Recent)**：保存近期只访问过一次的数据。
*   **T2 (Frequent)**：保存近期访问过至少两次的数据。
*   **B1 / B2 (Ghost Cache)**：幽灵缓存，仅记录从 T1/T2 淘汰的 Key。
*   **动态平衡**：通过动态参数 `p`（T1 目标容量），当命中 B1 时增大 `p`，命中 B2 时减小 `p`，实现对不同业务流量特征的自适应动态平衡。

### 2. Sharded Cache 分段锁路由
借鉴 `ConcurrentHashMap` 与 Memcached 思想，将全局缓存空间拆分为 16 个独立分片。
*   **路由规则**：`hash(key) % 16`
*   **优势**：每个分片维护独立的 ARC 实例与互斥锁，不同 Key 的并发读写大概率落在不同分片，极大提升吞吐量。

## 🚀 快速开始 & API 使用

项目启动后，默认通过 HTTP 协议暴露 RESTful API，监听 `8080` 端口。

### 写入缓存 (PUT)
```bash
# 语法: /put?key={key}&val={value}&ttl={ttl_in_seconds}
curl -X POST "http://localhost:8080/put?key=name&val=Kama&ttl=10"
```

### 读取缓存 (GET)
```bash
# 语法: /get?key={key}
curl "http://localhost:8080/get?key=name"
```

## 📊 性能测试 (Benchmark)

在 `test/` 目录下提供了多线程并发压测模块。
*   **测试环境**：10 线程并发，执行 100,000 次混合读写操作。
*   **测试结果**：

| 架构类型 | 10w 次读写耗时 | 并发表现分析 |
| :--- | :--- | :--- |
| **单全局锁版本** | `~1200 ms` | 锁争用极其严重，大量线程处于阻塞挂起状态，多核 CPU 优势无法发挥。 |
| **ShardARC (16分片)** | **`~350 ms`** | 锁竞争被分散到 16 个桶中，冲突极小，高并发吞吐能力极强。 |

**结论**：分段锁设计在多核并发场景下带来了 **约 3.4 倍** 的性能提升。同时，ARC 算法在应对带有局部热点和突发扫描的真实业务流量时，命中率显著优于纯 LRU。

## 🗺️ 未来规划 (Roadmap)

- [ ] **集群化扩展**：引入一致性哈希（Consistent Hashing）算法，支持多节点分布式集群部署。
- [ ] **数据持久化**：引入类似 Redis 的 AOF/RDB 持久化机制，保证进程宕机重启后的数据恢复。
-[ ] **内存淘汰优化**：增加内存占用监控，支持基于内存阈值（而非仅基于数量）的淘汰策略。
